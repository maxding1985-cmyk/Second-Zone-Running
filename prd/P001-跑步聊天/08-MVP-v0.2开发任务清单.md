# 跑步聊天 MVP v0.2 开发任务清单

> 创建日期: 2026-06-15  
> 版本目标: 把产品从“点击式流程原型”纠偏为“语音优先的二区说话测试 MVP”  
> 任务原则: 每个任务都必须服务于假设验证，不服务于核心假设的功能不进入本版本  
> 相关文档: `07-MVP-v0.2二区说话测试假设验证.md`

---

## 1. 版本范围

### 1.1 v0.2 必须交付的体验

用户完成一次 v0.2 测试跑时，应该发生这些事情：

1. 跑前理解什么是“二区/轻松有氧说话测试”。
2. 用户授权麦克风，知道 App 会在说话测试时录音。
3. App 用语音提示用户开始跑，不要求用户盯屏幕。
4. 跑中 App 至少 3 次发起说话测试。
5. 用户开口说完整句子，App 完成录音和转写。
6. App 根据用户说话内容判断：刚好、有点喘、过强、风险词或未知。
7. App 用语音反馈：保持、慢下来、走一会、停止运动安全提示。
8. 用户结束后看到围绕说话测试的报告。
9. 跑后问卷回答：是否更理解二区，是否觉得说话测试有帮助，是否想试真人陪跑。

### 1.2 本版本不交付

- 不做真人随机匹配。
- 不做 RTC 语音房。
- 不做手表心率接入。
- 不做社区/排行榜。
- 不做连续后台监听。
- 不做医疗级强度判断。

## 4. 强度分类 MVP 规则

### 4.1 输入输出

输入：STT 转写文本 + 录音时长 + 当前轮次。  
输出：分类结果 + 反馈类型 + 报告标签。

| 分类 | 输入样例 | 输出 | 正确反馈 |
| --- | --- | --- | --- |
| `target` | “我现在感觉还可以，呼吸有点快，但是能完整说话。” | 刚好 | “很好，保持这个节奏，不要急着加速。” |
| `too_easy` | “很轻松，我甚至可以唱歌。” | 偏轻松 | “可以保持轻松，也可以非常小幅提一点，但今天不追速度。” |
| `breathless` | “有点喘，说话要停一下。” | 偏强 | “慢一点，缩短步幅，先跑到能完整说话。” |
| `too_hard` | “说不完整，太累了，喘不过来。” | 过强 | “现在先降到走路，等呼吸恢复。” |
| `risk` | “我有点胸痛/头晕/心悸/恶心。” | 风险 | “请马上停止运动，必要时联系身边的人或专业医疗帮助。” |
| `unknown` | “嗯……不知道。” | 未知 | “没关系，我们再做一次：用一句完整的话描述呼吸。” |

### 4.2 风险词优先级

只要出现风险词，优先进入安全提示，不再继续普通陪跑：

```text
胸痛、胸闷、头晕、晕、眼前发黑、心悸、心慌、恶心、喘不过气、呼吸困难、刺痛、要晕倒
```

## 6. 验收用例速查

| 用例 | 输入 | 正确输出 |
| --- | --- | --- |
| 教育理解 | 用户跑前阅读说明 | 能看到“能说完整句子但不能唱歌” |
| 正常说话 | “我现在还可以，呼吸有点快但能完整说话” | 分类 `target`，语音提示保持 |
| 有点喘 | “我有点喘，说话要停一下” | 分类 `breathless`，语音提示慢下来 |
| 过强 | “我说不完整，喘不过来” | 分类 `too_hard`，语音提示走一会 |
| 风险词 | “我有点胸痛和头晕” | 分类 `risk`，立即安全停止提示 |
| STT 失败 | 空文本/识别失败 | 提示重试，不把本轮算作有效说话测试 |
| 不点按钮完成 | 用户只开始和结束，中间靠语音 | 报告能生成语音统计 |
| 真人兴趣 | 跑后选择“想试真人” | 后台记录 `human_companion_interest=yes` |

## 7. v0.2 完成定义

开发完成不等于 MVP 成功。v0.2 完成定义分两层：

### 7.1 工程完成

- Android APK 可以安装运行。
- 至少一种 STT/TTS 链路可用。
- 跑中至少 3 轮说话测试可完成。
- 后端记录语音事件。
- 报告展示说话测试结果。
- 自动化测试覆盖分类器和核心流程。

### 7.2 产品验证完成

- 至少 10 名用户完成真实跑步测试。
- 首次开口率 >= 60%。
- 多轮完成率 >= 40%。
- 主观有用率 >= 60%。
- 复用意愿 >= 50%。
- 得出“AI 足够 / 需要真人 / 方向不成立”的明确结论。

## 8. 推荐开发顺序

不要先做真人匹配，不要先做漂亮页面。建议顺序：

1. 先改说话测试教育和跑中脚本。
2. 接入 TTS，让 App 先能“说”。
3. 接入录音和 STT，让用户真的“说”。
4. 做规则分类和安全兜底。
5. 重构报告和数据指标。
6. 加自动化测试和真机测试用例。
7. 出 v0.2 APK，做小样本真实跑。
8. 再根据数据决定是否做真人陪跑匹配。
